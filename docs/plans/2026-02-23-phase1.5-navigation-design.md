# Phase 1.5 Design: Navigation Improvements

**Date**: 2026-02-23
**Status**: Approved
**Scope**: Incremental self-description enhancements to reduce agent navigation overhead

## Context

Phase 1 baseline (3 tasks, 100% correct) revealed navigation friction:
- Agent guesses `/sensor/sensor-1` IRI, fails, recovers via `SELECT DISTINCT ?sensor` to discover `/entity/sensor-1` (5 iterations vs 3 for tasks without this friction)
- VoID declares vocabularies but not entity URI patterns
- SHACL declares property constraints but not value types/patterns for entity references
- Only 2 SPARQL examples; agent must construct most queries from scratch

**Goal**: Reduce mean iterations and SPARQL attempts by enriching the self-description layers, measured incrementally against the Phase 1 baseline.

## Two URI Spaces

A fabric node operates across two distinct URI spaces. The design must keep them separate:

**Instance URI space** — local to this node:
- Pattern: `{base}/entity/{id}` (currently), eventually DID-anchored
- Contains: sensors, observations, samples, claims — actual data entities
- Declared via: `void:uriSpace` in VoID (L1)
- Dereferenceable: `GET /entity/{id}` → CONSTRUCT across named graphs

**Ontology URI space** — global vocabulary terms, cached locally:
- Examples: `http://www.w3.org/ns/sosa/`, `http://www.w3.org/2006/time#`
- Stored in: local TBox named graphs (`/ontology/sosa`, `/ontology/time`)
- The agent resolves ontology terms via local SPARQL, not web dereferencing
- This solves URI rot — vocabularies are available even if external URIs go down
- Declared via: `void:vocabulary` in VoID (L1), `sh:declare` prefix blocks in SHACL (L3)

**SHACL bridges the two spaces.** A property shape like:
```turtle
sh:property [
    sh:path sosa:madeBySensor ;     # ontology term (global, cached locally)
    sh:class sosa:Sensor ;          # ontology class (global, cached locally)
    sh:pattern "^{base}/entity/" ;  # instance URI pattern (local)
] ;
```
...maps from global vocabulary to local instance data. The agent learns: "this ontology property points to entities of this ontology type at this local URI pattern."

## Enhancement Layers

Four incremental enhancements, ordered by predicted impact (highest first). Each is measured independently against the Phase 1 baseline.

### Layer 1.5a: Entity Discoverability (VoID + SHACL)

**VoID** (`main.py` `_VOID_TURTLE`):
- Add `void:uriSpace "{base}/entity/"` — declares the instance entity namespace

**SHACL** (`shapes/endpoint-sosa.ttl`):
- Add `sh:declare` prefix block on the shapes graph — maps `sosa:`, `xsd:`, `time:` prefixes to their namespace IRIs, signaling which vocabularies are locally available
- Add `sosa:madeBySensor` property shape with:
  - `sh:class sosa:Sensor` — value type from ontology space
  - `sh:nodeKind sh:IRI` — values are IRIs, not literals
  - `sh:pattern "^{base}/entity/"` — values use the local entity namespace
  - `sh:description` — navigational hint text

**Discovery** (`fabric_discovery.py`):
- `_parse_void()` extracts `void:uriSpace` → new `FabricEndpoint.uri_space: str | None`
- `_parse_shapes()` extracts `sh:declare` prefix/namespace pairs and `sh:class`/`sh:pattern` from property shapes
- `routing_plan` renders entity URI space and ontology-to-instance mapping

**Experiment phase**: `phase1.5a-urispace`

**Predicted effect**: sensor-temp-by-name drops from 5 to ~3 iterations.

### Layer 1.5b: Named Graph Inventory

**VoID** (`main.py` `_VOID_TURTLE`):
- Declare available named graphs as `void:Dataset` partitions:
```turtle
void:subset [
    a void:Dataset ;
    dct:title "Observations" ;
    void:sparqlGraphEndpoint <{base}/graph/observations> ;
    dct:conformsTo fabric:ObservationShape ;
] ;
```

**Discovery** (`fabric_discovery.py`):
- `_parse_void()` extracts named graph inventory → new `FabricEndpoint.named_graphs: list[NamedGraphInfo]`
- `routing_plan` renders available graphs with their shapes and purpose

**Experiment phase**: `phase1.5b-graphinv`

**Predicted effect**: Agent learns graph names from VoID rather than relying solely on `sh:agentInstruction`. Moderate improvement — the agent hint already works, but this is more systematic.

### Layer 1.5c: Richer SPARQL Examples

**SPARQL examples** (`sparql/sosa-examples.ttl`):
Add 3 new examples:
1. **Discover sensors** — `SELECT DISTINCT ?sensor WHERE { GRAPH ... { ?obs sosa:madeBySensor ?sensor } }`
2. **Entity lookup** — `DESCRIBE <{base}/entity/{id}>` pattern (or CONSTRUCT equivalent)
3. **Count observations** — `SELECT (COUNT(?obs) AS ?n) WHERE { ... }`

**Experiment phase**: `phase1.5c-examples`

**Predicted effect**: Reduces exploratory queries. The sensor discovery example directly addresses the pattern the agent had to figure out independently.

### Layer 1.5d: Enhanced Routing Plan

**Discovery** (`fabric_discovery.py` `routing_plan` property):
Restructure the text rendering to present both URI spaces clearly:

```
Entity URI space: {base}/entity/
  Dereferenceable: GET /entity/{id} → CONSTRUCT across all named graphs

Local ontology cache (no external dereferencing needed):
  sosa: <http://www.w3.org/ns/sosa/> → /ontology/sosa
  time: <http://www.w3.org/2006/time#> → /ontology/time

Named graphs:
  /graph/observations — Observations (fabric:ObservationShape)

Shapes (1):
  ObservationShape → sosa:Observation
    sosa:madeBySensor → sosa:Sensor (entity IRI, pattern: {base}/entity/*)
    sosa:hasSimpleResult → xsd:double (literal)
    sosa:resultTime → xsd:dateTime (literal)

SPARQL Examples (5):
  ...
```

**Experiment phase**: `phase1.5d-routing`

**Predicted effect**: Cumulative improvement. Better presentation of all layers reduces cognitive overhead for the agent.

## Experiment Protocol

For each enhancement layer:

1. Apply the code changes
2. Update `PHASE_FEATURES` in `run_experiment.py` with the new phase identifier
3. Run the 3 existing baseline tasks (same task JSON, same scoring function)
4. Record: iterations, SPARQL attempts, empty recoveries, convergence, cost
5. Write JSONL trajectories for qualitative analysis
6. Compare against Phase 1 baseline quantitatively

**Phase feature mapping** (`run_experiment.py`):
```python
PHASE_FEATURES = {
    "phase1-baseline": ["void-sd", "shacl-agent-hints", "sparql-examples"],
    "phase1.5a-urispace": ["void-sd", "void-urispace", "shacl-prefixes", "shacl-class-pattern", "shacl-agent-hints", "sparql-examples"],
    "phase1.5b-graphinv": ["void-sd", "void-urispace", "void-graph-inventory", "shacl-prefixes", "shacl-class-pattern", "shacl-agent-hints", "sparql-examples"],
    "phase1.5c-examples": ["void-sd", "void-urispace", "void-graph-inventory", "shacl-prefixes", "shacl-class-pattern", "shacl-agent-hints", "sparql-examples", "sparql-examples-extended"],
    "phase1.5d-routing": ["void-sd", "void-urispace", "void-graph-inventory", "shacl-prefixes", "shacl-class-pattern", "shacl-agent-hints", "sparql-examples", "sparql-examples-extended", "enhanced-routing-plan"],
}
```

## Success Criteria

- Each layer shows measurable improvement on at least one metric (iterations, SPARQL attempts, or empty recoveries)
- sensor-temp-by-name drops from 5 iterations to ≤3 after 1.5a
- No regressions on tasks that already pass in ≤3 iterations
- All existing tests continue to pass
- JSONL trajectories available for qualitative comparison

## Non-Goals

- No write-side tools (Phase 2)
- No DID/VC identity (Phase 2)
- No new task definitions (we measure improvement on the same 3 tasks)
- No changes to Docker infrastructure or Oxigraph config
