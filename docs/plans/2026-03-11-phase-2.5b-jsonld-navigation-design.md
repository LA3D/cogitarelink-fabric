# Phase 2.5b: JSON-LD Hypermedia Navigation Experiments — Design

**Date**: 2026-03-11
**Prerequisite**: Phase 2.5a complete (JSON-LD conneg, `@context` files, `/ontology/{vocab}` routes)
**Harness**: `experiments/node-rlm-fabric/` (node-rlm + Anthropic driver)

---

## Research Question

Where is the boundary between what JSON-LD navigation can answer vs. when the agent needs SPARQL? Does JSON-LD vocabulary navigation help agents construct better SPARQL queries against self-describing endpoints?

This is a co-evolutionary experiment: agent behavior reveals fabric structure gaps, we fix the structure, re-run. The experiments measure the fabric's affordance design as much as the agent's capability.

## Design Philosophy

General-purpose primitives, not task-specific helpers. Give the agent foundational libraries for JSON-LD processing and SPARQL querying, let it compose them programmatically in the RLM sandbox. Don't over-engineer — observe where it falls over, iterate.

---

## New Tools

### `fetchJsonLd(url: string): Promise<string>`

General-purpose HTTP GET with `Accept: application/ld+json`. Uses existing `createFabricFetch` (VP Bearer auth wrapper). Works for fabric URLs (`/ontology/sosa`, `/.well-known/void`) and external vocab URIs. Returns raw JSON-LD string. Errors return readable messages.

### `jsonld` namespace (sandbox global)

Expose `jsonld` (v8.3.0, already in package.json) operations:

- `jsonld.expand(doc)` — Resolve prefixes, make all IRIs absolute
- `jsonld.compact(doc, context)` — Compact against a context
- `jsonld.frame(doc, frame)` — Extract subgraph matching a frame pattern (structured queries without SPARQL)

All accept parsed JS objects. Agent workflow: `JSON.parse(await fetchJsonLd(url))` then `jsonld.*` operations.

### No changes to existing tools

`comunica_query`, `fetchVoID`, `fetchShapes`, `fetchExamples`, `fetchEntity` remain as-is.

## Experiment Conditions

| Condition | Tools Available | Tests |
|-----------|----------------|-------|
| `js-baseline` | comunica_query, fetchVoID, fetchShapes, fetchExamples, fetchEntity | SPARQL-first navigation (existing results from 2026-03-06) |
| `js-jsonld` | comunica_query, fetchJsonLd, jsonld.*, fetchVoID, fetchShapes, fetchExamples, fetchEntity | JSON-LD for vocabulary understanding, SPARQL for data retrieval |
| `js-combined` | All tools | What does the agent reach for? When does it switch? |

**Key design choice**: `js-jsonld` keeps `comunica_query` for data retrieval. The hypothesis is that JSON-LD vocabulary navigation produces better SPARQL queries, not that SPARQL can be eliminated. The `globalDocs` guide the agent toward `fetchJsonLd`/`jsonld.*` for schema exploration.

## Task Design

### Category 1: Existing SIO tasks (6 tasks, reused)

Direct comparison across all three conditions. Baseline scores 1.0, so these measure efficiency differences (iterations, SPARQL attempts, recoveries) more than correctness.

Task file: `tasks/phase1-sio-baseline.json`

### Category 2: Vocabulary navigation tasks (3-4 new tasks)

Require understanding ontology structure not spelled out in SPARQL examples or SHACL shapes. The agent must reason about the vocabulary itself.

| Task | What It Forces | Why JSON-LD Might Help |
|------|---------------|----------------------|
| Property chain: Observation → Sensor | Navigate SOSA madeBySensor domain/range | `jsonld.frame` for `rdfs:domain`/`rdfs:range` |
| Subclasses of sosa:FeatureOfInterest | Query across ontology named graphs for `rdfs:subClassOf` | `jsonld.expand` makes class hierarchy explicit |
| SIO properties with xsd:float range | SIO schema introspection (no SPARQL example) | Expand SIO ontology, filter by range in code |
| Inverse chain: MeasuredValue → Observation | Multi-hop through SIO + SOSA property axioms | Frame for `owl:inverseOf`, compose chain |

These tasks are designed to surface the boundary — where SPARQL exploration is weakest (guessing predicates, unbounded scans) and JSON-LD expansion makes full schema visible as navigable data.

Task file: `tasks/phase2.5b-vocab-navigation.json`

These are starting hypotheses. Task design will evolve based on observed agent behavior.

## Measurement

### Harness captures (per-task, automatic)

- Score (substring match)
- Iterations to answer
- Wall time, token usage (prompt/completion), cost
- Full trajectory via `RlmObserver` events (every code block + output)

No new metrics in the harness — keep it simple.

### Prose agent-as-judge analysis (post-experiment)

A prose file (`analyze-phase2.5b.prose`) performs deep trajectory analysis:

- **Tool choice classification**: Per iteration, which tool family (JSON-LD vs SPARQL)?
- **Navigation strategy**: JSON-LD first then SPARQL? SPARQL exploration? Mixed?
- **Vocabulary understanding depth**: How many expand/frame calls before correct query?
- **The boundary moment**: When and why did the agent switch strategies?
- **Recovery patterns**: Failed approach → successful fallback

Template: adapt existing `analyze-trajectories.prose` with Phase 2.5b-specific classification agents.

### Reporting

- Per-condition comparison table (score, iterations, cost)
- Tool choice heatmap: task × tool (which tool for which task)
- Qualitative trajectory analysis on boundary cases

### js-jsonld

```
You have access to JSON-LD processing tools for understanding vocabulary structure.

fetchJsonLd(url) — Fetch any URL as application/ld+json. Returns raw JSON-LD string.
jsonld.expand(doc) — Expand compact JSON-LD to explicit form (all IRIs absolute).
jsonld.compact(doc, context) — Compact expanded JSON-LD against a context.
jsonld.frame(doc, frame) — Extract matching subgraph using a JSON-LD frame.

Discovery strategy:
1. Use fetchVoID/fetchShapes/fetchExamples for endpoint self-description
2. Use fetchJsonLd to retrieve vocabulary definitions from /ontology/{vocab}
3. Use jsonld.expand/frame to understand property domains, ranges, and class hierarchies
4. Construct SPARQL queries informed by vocabulary structure
5. Use comunica_query to execute queries against data graphs
```

## Success Criteria

- `fetchJsonLd` and `jsonld.*` operational as RLM sandbox callables
- Three conditions runnable: js-baseline, js-jsonld, js-combined
- Existing SIO tasks: js-jsonld and js-combined match js-baseline score within 0.1
- New vocab tasks: js-jsonld scores above 0 (proves JSON-LD navigation works at all)
- Trajectory data captured for all conditions
- Prose analysis produces tool-choice classification and boundary findings

## Non-Goals

- Non-fabric JSON-LD source experiment (deferred — validate tools on fabric first)
- Custom metrics in the harness (use prose post-analysis instead)
- Production-quality tools (iterate based on observed agent behavior)
